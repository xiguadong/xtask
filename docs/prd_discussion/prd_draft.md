# PRD 草案

## 1. 执行摘要
- 问题陈述：当前需要一款可在浏览器访问的项目管理产品，用于提升任务协作、进度透明与工程化管理体验。
- 目标交付物：一版 Web 项目管理应用的 MVP 需求定义、功能边界、信息流与可执行验收标准。
- 成功标准：不建设 KPI 系统；采用“关键任务链路功能门槛”作为发布验收标准。

## 2. 用户与产品形态
- 目标用户：个人用户（已确认）。
- 核心场景：个人项目下的任务拆解、统一状态流转、里程碑追踪、进度复盘。
- 交互载体（硬件/软件）：浏览器 Web 应用（桌面优先，兼容移动端访问）。
- GitHub 对齐优先级：`Milestones > Projects(Board/List) > Issues(Task)`；MVP 不包含 Discussions。

## 3. 功能需求
- 核心功能：
  - 项目空间与任务管理（Task 创建、编辑、统一状态流转）。
  - 看板与列表视图（按状态、优先级、截止时间组织）。
  - Milestones 管理（目标、截止时间、完成进度）。
  - 基础任务检索与筛选（快速定位任务）。
  - 任务关系表达（父子、依赖、关联/耦合、独立）。
- 支持功能：
  - 标签、优先级、排序与搜索。
  - 任务备注与变更历史（个人追踪用途）。
- 可选功能：
  - 评论/@提及/通知（当扩展到多人协作时再启用）。
  - 时间线/日历视图、外部系统集成。
- 非目标：
  - 不做多人协作通知。
  - 不做外部系统集成。
  - 不做自动化规则。

## 3.1 任务关系模型（MVP）
- 关系类型：
  - `Parent-Child`：用于模块/功能拆解，形成任务树。
  - `Blocks` / `Blocked-By`：用于前后置依赖，形成有向关系。
  - `Related-Strong`：用于强耦合关系，需在详情页显著提示联动影响。
  - `Related-Weak`：用于弱耦合关系，仅用于关联追踪。
  - `Independent`：默认关系，任务互不关联。
- 设计原则：
  - 一个任务最多 1 个父任务，可有多个子任务。
  - 依赖关系允许多对多，但禁止循环依赖。
  - `Blocked-By` 为强约束：存在未解除阻塞时，任务不能进入 `已完成`。
  - `Related-Strong` 与 `Related-Weak` 均不作为状态机强约束，但要在任务详情中可区分展示。
  - 看板卡片展示关键关系徽标，详情页展示完整关系列表。

## 3.2 新建任务流程（MVP）
- 创建入口：
  - 看板页主按钮 `New Task`（默认入口）。
  - 任务详情页 `Add Subtask`（继承父任务上下文）。
- 创建方式：
  - 快速创建：填写必填字段后立即创建。
  - 高级创建：在同一表单中附加父任务、依赖、关联关系（可选强耦合/弱耦合）。
- MVP 必填字段：
  - 标题、描述、状态、优先级、截止日期、里程碑。
- 默认值策略：
  - 状态默认 `待办`。
  - 若从父任务下创建，自动继承父任务里程碑（可修改）。
  - 关系未填写时默认为 `Independent`。

## 3.3 轻量化数据存储与维护（MVP）
- 存储方式：
  - 采用文件化存储，不引入数据库。
  - 每个项目目录维护一个任务图文件：`<project_dir>/.agents/task_graph.yaml`。
- 文件职责：
  - 记录项目元信息、里程碑、任务字段、任务关系（父子/依赖/耦合）。
  - 作为 AGENTS 的唯一写入源，保证可追踪与可审阅。
- 写入策略：
  - 读-改-写单文件，使用临时文件替换方式避免写坏（`tmp -> rename`）。
  - 每次写入更新 `updated_at` 和 `updated_by`（标记为 `agent`）。
- 约束边界：
  - MVP 不做跨项目聚合存储。
  - MVP 不做复杂同步冲突合并，仅保证单项目单工作流轻量可用。

## 3.4 功能地图（阶段 B）
- 核心功能（必须有）：
  - 项目空间与任务管理（统一状态机）。
  - 看板与列表视图（看板默认首页）。
  - Milestones 管理与进度聚合。
  - 任务关系建模（父子、依赖、强/弱耦合、独立）。
  - 新建任务双模（快速/高级）。
- 支持功能（提升体验）：
  - 标签、优先级、筛选、排序、搜索。
  - 任务备注与操作历史。
- 可选功能（后续迭代）：
  - 评论/@提及/通知（多人协作阶段再启用）。
  - 时间线/日历视图。
- 非目标（明确不做）：
  - 外部系统集成。
  - 自动化规则引擎。
  - 多人协作通知系统。

## 3.5 隐含步骤回放（阶段 C）
- 核心任务 1：快速创建任务
  - 用户先看到什么：看板页顶部 `New Task` 按钮。
  - 用户接着做什么：输入必填字段并提交。
  - 系统给出什么反馈：任务卡片出现在 `Todo` 列并写入 `task_graph.yaml`。
  - 到什么状态算完成：卡片可见、字段可查、文件落盘成功。
- 核心任务 2：建立父子/依赖关系
  - 用户先看到什么：任务详情中的关系设置区。
  - 用户接着做什么：选择父任务或依赖任务并保存。
  - 系统给出什么反馈：详情关系列表更新，看板卡片显示关系徽标。
  - 到什么状态算完成：关系可追溯且通过规则校验（无循环依赖）。
- 核心任务 3：处理阻塞任务
  - 用户先看到什么：任务状态与阻塞关系提示。
  - 用户接着做什么：尝试将任务设为 `已完成`。
  - 系统给出什么反馈：若存在未解除阻塞，拒绝完成并给出原因。
  - 到什么状态算完成：阻塞解除后才允许进入 `已完成`。
- 核心任务 4：里程碑进度追踪
  - 用户先看到什么：里程碑详情与关联任务列表。
  - 用户接着做什么：推进任务状态并刷新视图。
  - 系统给出什么反馈：里程碑完成进度同步变化。
  - 到什么状态算完成：进度与任务状态一致且可复现。

## 4. 关键信息流
- 输入：用户创建任务、设置任务关系、更新状态、设置里程碑、执行筛选查询。
- 转换：系统进行字段校验、关系校验（父子/依赖/循环）、状态流转校验、里程碑聚合与检索计算。
- 输出：任务状态变化、关系可视标记、看板/列表结果、里程碑进度、操作历史。
- 反馈闭环：用户根据关系与进度结果继续拆分任务、调整依赖并推进完成。

## 4.3 UI-UX Pro Max 设计细则（MVP）
- 信息层级原则：
  - 首页只回答“有哪些项目、哪个项目有风险、下一步做什么”。
  - 子页只回答“当前项目里要做哪些任务、任务关系如何、我下一步点哪里”。
  - 详情信息采用按需展开，不默认常驻占据主视图。
- 交互原则：
  - One Click Forward：用户每次操作后必须有明确下一步入口（如 `New Task`、`Open Detail`、`Resolve Block`）。
  - Local First Context：子页操作默认在当前项目内，不跨项目跳转。
  - Progressive Disclosure：默认展示核心字段，关系/历史/高级项按需展开。
- 过滤与检索：
  - 统一过滤条：状态、优先级、里程碑、标签、截止时间、是否阻塞。
  - 支持保存过滤视图（Saved Filters），并在左侧快速切换。
  - 过滤条件变化后，URL/状态可复现，便于回放与自检。
- 任务关系可读性：
  - 卡片显示关系徽标：`P`、`C:n`、`B:n`、`BB:n`、`R!`、`R~`。
  - 详情区展示关系列表（上游/下游/耦合），点击任一关系可定位并高亮目标任务。
  - 阻塞链必须可追溯到根因任务（至少 1 跳可见）。
- 左右联动规则（子页）：
  - 左侧筛选切换时，主区与详情区同时更新；若当前详情任务被过滤掉，详情区提示并可一键清除过滤。
  - 主区卡片被选中时，详情区同步打开并聚焦字段。
  - 详情区更新状态/关系后，主区卡片与左侧统计即时刷新。
- 新建任务体验：
  - 快速创建默认打开，字段最小集一次录入即可提交。
  - 高级项（关系、父任务、耦合等级）采用折叠区，避免首次认知负担。
  - 创建成功后自动定位到新卡片并短暂高亮，确认“已生效”。
- 状态评价体验：
  - 执行状态：`Todo / Doing / Blocked / Done`。
  - 健康状态：`Healthy / At Risk / Blocked`，支持系统建议与人工修正。
  - 当状态被规则拦截（例如存在 `Blocked-By`）时，必须给出可执行修复提示。
- 反馈与空状态：
  - Loading：使用骨架屏，避免布局抖动。
  - Empty：给出“下一步动作”按钮（如 `Create First Task`）。
  - Error：提示原因、影响范围、重试按钮，不隐藏上下文。
- 无障碍与键盘效率：
  - 关键操作支持键盘快捷键（示例：`N` 新建任务，`/` 搜索）。
  - 颜色不作为唯一状态表达，必须有文本/图标辅助。
  - 关键交互可在 200% 缩放下保持可读与可操作。

## 4.5 信息流定稿（阶段 D 完成）
- 一级首页信息流：
  - 输入：项目筛选（状态/里程碑/关键词）。
  - 转换：项目状态聚合、风险分类、摘要统计计算。
  - 输出：项目总览看板、Summary Panel、可进入子页的项目卡片。
  - 反馈闭环：用户根据风险分布进入项目子页执行任务推进。
- 二级子页信息流：
  - 输入：任务筛选、任务编辑、关系维护、状态流转。
  - 转换：字段校验、关系校验、阻塞规则校验、里程碑进度重算。
  - 输出：Board/List 任务视图、详情抽屉、关系高亮、里程碑进度。
  - 反馈闭环：用户处理阻塞与依赖后继续推进任务到完成。

## 5. 验收与测试
- 指标与阈值：采用功能门槛阈值，不采用运营 KPI（例如：关键流程一次通过、状态流转一致性）。
- 测试方法与场景：功能测试 + 端到端用户任务回放（个人用户场景）。
- 验收证据：录屏、操作日志、测试报告（不要求 KPI 仪表盘截图）。

## 5.1 界面布局与风格一致性验收（已定稿）
- 目标：验证最终实现界面与讨论确认的布局/风格基线一致。
- 验收机制：
  - 第一层：系统自检（自动执行，覆盖大部分功能与布局一致性）。
  - 第二层：人工终检（由人做最终风格确认与例外裁决）。
- 自检机制（自动）：
  - 页面结构自检：按页面与区域清单校验 DOM/组件存在性（首页一级 + 子页二级）。
  - 关键流程自检：新建任务、进入子页、打开详情、状态流转、里程碑更新。
  - 截图对照自检：固定视口自动截图并与基线进行结构比对。
  - 固定视口：桌面 `1366x768`，移动 `390x844`。
- 检查清单（MVP）：
  - 首页（一级）必须展示项目概要看板，且包含项目状态分区。
  - 子页（二级）必须展示项目任务看板，且包含四列状态（Todo/Doing/Blocked/Done）。
  - 桌面端一级与二级页面均需存在 Top Bar、Sidebar、Main Area、Detail Area（或同等分区）。
  - 任务详情在桌面端可右侧展开，在移动端可全屏打开。
  - 任务卡片必须展示标题、优先级、截止日期、里程碑标识。
  - `New Task` 在首页与子页、桌面与移动均可单击直达创建流程。
- 通过标准（预设）：
  - 自动自检覆盖项通过率 >= 90%，且关键路径项通过率 = 100%。
  - 布局结构阻塞偏差 = 0（首页一级、子页二级都满足）。
  - 人工终检通过：风格一致性与信息层级通过最终确认。

## 5.2 UI-UX 自检清单（自动）
- 页面结构自检：
  - 首页存在项目总览核心区；子页存在任务主视区与详情交互区。
  - 桌面与移动在固定视口下关键区域均可见。
- 行为自检：
  - 过滤条件可叠加、可清除、可复现。
  - 新建任务后任务立即出现在当前视图并落盘。
  - 关系跳转可定位目标任务并高亮。
  - 阻塞规则触发时有明确错误提示与修复入口。
- 视觉自检：
  - 状态颜色与文本标识同时存在。
  - 关键按钮在不同状态（默认/悬停/禁用）下对比可辨识。
- 可访问性自检：
  - 主流程可仅使用键盘完成。
  - 关键控件具备可读标签与焦点顺序。
